System Architecture & Docs

A comprehensive deep-dive into the modules, workflows, and operations of the IPEC Expense Manager web application ecosystem.

Command Center Overview admin.html

Dashboard & Analytics Engine
The central hub for administrative oversight. It pulls vast amounts of expense data across the organization in real-time, displaying crucial metrics like "Total Corporate Spending" and "Pending Approvals." This module is designed to provide immediate situational awareness using visual tools (via Chart.js) and quick KPI cards.

Usage: Financial Controllers and Main Admins use this dashboard to instantly verify the health of the company's financial burn rate over specific quarters or months.

Logical Workflow Execution

1
Real-time Listener Initialization: The page orchestrates a Firebase onSnapshot listener binding to the expenses and users collections immediately after the admin authenticates successfully.
2
Data Aggregation Pipeline: Incoming documents are mapped and reduced locally. Variables accumulator adds up amounts where status === 'Approved'. Pending requests are pushed into a separate review array.
3
Chart Data Transformation: The reduced dataset is split into labels (Dates, Categories, Departments) and datasets (Amounts). This transformed data is fed directly into the Canvas contexts replacing old Chart.js instances.
4
DOM Repaint: Counter elements animating from 0 to the target total are triggered. The UI shifts to resolve loading states, displaying the latest financial state of the firm securely.
Firebase Firestore onSnapshot() Chart.js new Chart() requestAnimationFrame for Counters
Expense Evaluation & Adjudication Module
A critical compliance gateway. It lists all pending claims submitted by the employee workforce. Admins review itemized lines, inspect uploaded receipts, and make definitive binary decisions (Approve/Reject) or push for revisions.

Usage: Used daily by HR managers or department heads to authorize reimbursements. Requires scrutiny of PDF/Image proofs against stated request amounts.

Logical Workflow Execution

1
Queue Construction: The UI builds a table or grid of items where status === 'Pending', sorted by timestamp descending using a Firestore query with orderBy('createdAt', 'desc').
2
Deep Inspection Trigger: Admin clicks "Review". A modal expands capturing the specific document via its ID. It triggers a fetch for the attached receiptUrl from Firebase Cloud Storage securely.
3
Adjudication Transaction: If accepted, the Admin clicks "Approve". An asynchronous transaction begins. The system records the processedBy (Admin UID) and processedAt (Server Timestamp).
4
Atomic Write: The updateDoc() command is issued changing status to 'Approved'. This triggers a history update locally where a new entry is pushed into the history array of the document.
query(collection(db, 'expenses'), where('status', '==', 'Pending')) updateDoc(docRef, { status: 'Approved' })
WebRTC Connectivity (Video & Voice)
A cutting-edge peer-to-peer communication system integrated directly into the Admin Portal. It allows Admins to initiate instant Video/Voice calls with employees to clarify expense details without leaving the dashboard.

Usage: Vital for resolving complex claim disputes or conducting quick internal audits where high-bandwidth communication is preferred over asynchronous chat.

Technical Signaling Workflow

1
SDP Offer Generation: The caller initiates RTCPeerConnection and creates an SDP (Session Description Protocol) offer, which is written to a dedicated calls collection in Firestore.
2
Signaling Implementation: Both peers listen to Firestore sub-collections (offerCandidates and answerCandidates). ICE candidates are swapped in real-time as they are discovered by the browser's networking stack.
3
Media Stream Negotiation: Upon answer acceptance, localStream (from getUserMedia) is attached to the peer connection, and ontrack events trigger to display the remoteStream in the UI's video elements.
4
Call Termination: Either peer can trigger cleanupCall(), which closes the connection, stops all tracks, and updates the Firestore call document status to 'ended'.
new RTCPeerConnection() navigator.mediaDevices.getUserMedia() onSnapshot(collection(callRef, 'candidates'))
Global Notification Pipeline
The heartbeat of system-wide communication. Admins can dispatch targeted notifications to individual users or broadcast emergency bulletins to the entire organization simultaneously.

Usage: Used for system maintenance alerts, expense policy changes, or notifying employees of completed payments.

Dispatch Logic

1
Recipient Targeting: Admins select a specific User ID or the reserved 'ALL' keyword to trigger a broadcast.
2
Document Dispatch: A new document is created in the notifications collection with a read: false flag and a server-side timestamp.
3
Listener Perception: Employee portals (listening to their specific UID or 'ALL') receive the new data packet via onSnapshot and trigger a local UI 'Toast' or badge update.
addDoc(collection(db, 'notifications'), payload) onSnapshot(query(where('targetUserId', 'in', [uid, 'ALL'])))
Financial Auditing & Reporting Engine
A robust reporting suite that allows financial controllers to generate immutable PDF artifacts of expense data. It precisely maps Firestore documents to a high-fidelity DOM structure optimized for PDF rendering.

Usage: Used at the end of every fiscal month or project milestone to create official records for corporate tax and internal auditing.

Audit Generation Lifecycle

1
Data Vectorization: The engine queries the expenses collection using date-range filters and aggregates totals by Department and Category into a JSON-serialized object.
2
Virtual DOM Mapping: A hidden <div> is populated with a stylized template containing the filtered data, including auto-calculated KPI totals and high-resolution charts.
3
PDF Serialization: Using html2pdf.js, the engine captures the virtual DOM and converts it into a multi-page PDF blob with high DPI settings (300+ DPI) for professional print quality.
html2pdf().from(element).save() JSON.stringify(auditPayload)
Real-time Session Management
Ensures that the Admin Portal stays synchronized with the latest workforce state. It manages user authentication persistence, role verification, and global state synchronization.

Usage: Critical for security; it ensures that any changes to user permissions or budget limits are reflected in the UI across all active admin terminals instantly.

Security & Sync Workflow

1
Auth Listener: onAuthStateChanged initializes the session, fetching the user's document from the users collection to verify the ADMIN role.
2
RBAC Guard: Every sensitive function checks userData.role before execution, preventing unauthorized API calls at the frontend layer.

admin.html - Architecture & Tech Stack

The Admin portal is built as a highly responsive, client-side rendered Single Page Application (SPA) utilizing vanilla HTML/JS paired with modern utility CSS framing.

Styling Core

Tailwind CSS via CDN. Utilizes extensive utility classes for rapid layout building, flexbox structuring, glassmorphism UI components, and strict responsive breakpoint management natively handling dark/light mode switches.

Database & Auth

Firebase V9 (Modular). Incorporates Firestore for NoSQL document syncing, Firebase Storage for receipt blob holding, and Firebase Authentication for strict session lifecycle management guarding admin routes.

Data Visualization

Chart.js CDN. Used for drawing performant canvas-based line, bar, and doughnut charts for financial analytics, making data digestable immediately.

Exporting Engine

html2pdf.js / CSV Parsing. Includes client-side bundling libraries that capture the DOM or map JSON arrays to export comprehensive audit reports instantly without server overhead.

Employee Workspace emp.html

AI-Powered Submission Portal
A revolutionary interaction layer that leverages Google Gemini (via ai-support.js) to automate expense creation. Users can simply chat with the AI, which parses natural language to extract project codes, amounts, and categories.

Usage: Significantly reduces friction for staff on-the-go. Instead of manual data entry, they describe their expense, and the AI drafts the submission for them.

AI Logic Flow & Integration

1
Natural Language Parsing: The AISupport class sends user prompts to the Gemini API with a system instruction to return a specialized [COMMAND:CREATE_EXPENSE:...] tag if all data points are present.
2
Event Dispatch: Upon receiving the command, ai-support.js dispatches a CustomEvent('ai-expense-action') globally.
3
Autonomous Modal Handling: emp.html listens for this event and invokes window.createExpenseFromAI(data), which programmatically opens the submission modal, populates all fields, and triggers the submitExpense() function after a brief delay.
window.dispatchEvent(new CustomEvent('ai-expense-action')) Gemini API: generateContent() window.createExpenseFromAI(payload)
PWA & Offline Resilience
Designed as a Progressive Web App, emp.html ensures that employees can access their dashboard and draft expenses even in low-connectivity environments like airplanes or remote sites.

Usage: Enables "Install as App" on mobile devices and ensures critical UI assets are cached locally for near-instant rendering.

Lifecycle Management

1
Service Worker Registration: navigator.serviceWorker.register('./sw.js') is called on page load, installing a background worker that runs independently of the main browser thread.
2
Cache-First Strategy: The sw.js intercepts fetch requests and retrieves core assets (CSS, Icons, Fonts) from the CacheStorage, falling back to the network only when necessary.
3
Manifest Integration: High-resolution icons and theme colors are defined in manifest.json, allowing the browser to prompt users to "Add to Home Screen" for a native, app-like feel.
caches.match(event.request) navigator.serviceWorker.register() ServiceWorkerRegistration.showNotification()

Usage: Critical for employees to verify the timeline of their reimbursement and view granular feedback from the finance department if a claim requires revision.

3
Badge Color Mapping: The UI utilizes a specific CSS map to apply status-specific styling (e.g., bg-amber-100 for PENDING_TREASURY) ensuring immediate visual hierarchy.

emp.html - Architecture & Tech Stack

The Employee portal relies on the same robust foundations but is explicitly tuned to prevent wide database read access, prioritizing strict UX boundaries, form handling, and speed.

Responsive DOM Processing

Vanilla JavaScript Events. The UI is manipulated via standard DOM node manipulation mapped to form inputs, preventing heavyweight framework load times for standard operators.

Access Security

Firebase Security Rules matching. Queries from emp.html implicitly expect the backend rules to challenge them, making sure no client-tampering can ever retrieve data belonging to another UID.

Styling Core (Light Mode Default)

Tailwind CSS Utilities. The portal defaults to a brighter, highly legible color spectrum utilizing `bg-slate-50` and structured borders for clear form interactions, distinguishing it aesthetically from the dark mode command center.

PWA Ready

Progressive Web App Logic. Employee workflows incorporate `manifest.json` parsing and Service Worker bindings allowing staff to "Install" the portal as a native app on their phones for quick photo receipt uploads.
